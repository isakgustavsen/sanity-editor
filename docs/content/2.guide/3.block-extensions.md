## Why block extensions?

Standard paragraphs, headings, lists, and links are handled by the built-in transformers. For **custom block types** (e.g. tasks, callouts, embeds), you add:

1. **TipTap extensions** that define the node types in the editor.
2. A **`SanityEditorBlockExtension`** object that tells the module how to convert between TipTap/ProseMirror JSON and your block `_type` (Portable Text shape).

## `SanityEditorBlockExtension`

This is a **union** of two shapes:

### `mode: 'single'`

One TipTap top-level node maps to one or more blocks.

- **`type`** — Block `_type` used when serializing (Portable Text `_type`).
- **`tiptapNodeTypes`** — TipTap node type names this extension handles (e.g. `['taskItem']`).
- **`fromTiptapNode`** — TipTap node → `PortableTextBlock[] | null`.
- **`toTiptapNode`** — One `PortableTextBlock` → TipTap node JSON or `null`.
- **`schemaDefinition`** (optional) — Fragment merged into the schema (`blockObjects`, etc.).

### `mode: 'runContainer'`

A **container** node holds a **run** of items; several consecutive blocks of the same `_type` map to one container.

- **`containerTiptapNodeType`** — e.g. `taskList`.
- **`itemTiptapNodeType`** — e.g. `taskItem`.
- **`fromTiptapNode`** — Container node → `PortableTextBlock[] | null`.
- **`toTiptapNodeRun`** — Given `blocks` and `startIndex`, return `{ node, nextIndex }` or `null` to skip.
- **`schemaDefinition`** (optional) — Same as above.

When **`blockExtensions`** is non-empty, the base **`sanityEditorDefaultSchemaDefinition`** is merged with each extension’s **`schemaDefinition`** (deduped by name) and `compileSchema` is used unless you pass an explicit **`schema`**.

## Example: task list (playground)

The repo’s **`playground/taskBlockExtension.ts`** defines a **`runContainer`** extension for **`task`** blocks:

- TipTap: `taskList` containing `taskItem` nodes.
- Stored blocks: consecutive `_type: 'task'` blocks with `checked` and `text` fields.

It implements **`fromTiptapNode`** (walk `taskList` → emit `task` blocks) and **`toTiptapNodeRun`** (consume a run of `task` blocks → one `taskList` node). Use that file as a reference when you add your own TipTap + block pairing.

## Keys

Use **`keyGenerator`** on the component or composable so new blocks get stable **`_key`** values during conversion (see `SanityEditorTransformContext` in the utilities API).
